Wouldn't it be great if you could change a whole interface palette by moving
one `hue` slider?

Maybe add a `saturation` slider on top of that, just to decide how colorful the
system should feel.

I'm not talking about a beautiful gradient or a decorative palette. I mean the
boring parts of an interface: backgrounds, text, borders, hover states, selected
states, disabled states, shadows, dark mode, and high contrast.

That sounds like it should be simple. Pick a hue, pick a saturation, and derive
the rest.

Then you try to make it work for every hue.

I have been chasing that idea for years, through several color spaces. Most of
the problems along the way were fixable. The one that took me longest to notice
was a wrong assumption baked into my own mental model:

**lightness is not contrast.**

This post is the story of finding that bug — and of OKHST, the small color
space I built to fix it.

## The Old Attempt

About seven years ago, I tried to make UI color generation happen at runtime.

The idea was simple: instead of hand-picking every token in a design system, the
framework should generate useful colors from a smaller set of inputs. Give it a
hue and saturation, then derive the rest: text, surfaces, borders, accents,
states, dark mode, and contrast variants.

This became possible because I found [HSLuv](https://www.hsluv.org/).

HSLuv gave me something regular HSL did not: a perceptually useful lightness
axis. In regular HSL, `50%` lightness can mean wildly different things for
yellow, blue, red, and green. HSLuv made "make this lighter" and "make this
darker" much closer to operations a UI system could rely on.

I spent months tuning formulas around it. Eventually the result was good enough
for production use. It was one of the most fun things I had built at the time,
because every change was visible immediately.

The old demo is still online: [theme.tenphi.me](https://theme.tenphi.me)

{/* TODO: screenshot — old HSLuv demo at theme.tenphi.me (not final) */}

But the system had limits.

It was deeply coupled to one design system. It was integrated into the framework
we used. The formulas were tuned for one product. HSLuv itself was also still an
immature color space, and I had to work around inconsistencies locally.

So even though the idea worked, it did not become a reusable tool. I kept using
HSLuv in projects but stopped shipping generated palettes, though I never
stopped thinking that UI colors should be computable.

## The Web Caught Up

The interesting part is that the web changed.

Today we have OKLCH in CSS. That is a huge step forward. It gives the platform a
modern perceptual color space, and it makes many color operations more sensible
than HSL ever could.

But OKLCH is still hard to use directly as a UI palette generator.

**The issue is chroma.**

In OKLCH, `C` is absolute chroma. It is not the same thing as the kind of
"saturation" control people expect when they author UI colors. The maximum
possible chroma changes depending on hue and lightness. If you keep chroma fixed
while changing lightness, some colors can leave the sRGB gamut and get clipped.

```css
/* Same chroma, different lightness.
   Some hue/lightness combinations may clip in sRGB. */
.step-1 {
  color: oklch(35% 0.18 250);
}
.step-2 {
  color: oklch(65% 0.18 250);
}
.step-3 {
  color: oklch(85% 0.18 250);
}
```

This is manageable if you are ready to define dozens of colors manually. You can
pick every value by eye and adjust the outliers.

The problem appears when colors need to be related to each other, like in a
dynamic theming system. If a background, text color, border, shadow, and hover
state are all derived from one source color, clipping silently changes the
relationship you were trying to preserve. A generated ramp can look fine in the
middle and fall apart near the edges.

Modern CSS makes this very tempting:

```css
.card {
  --bg: oklch(65% 0.18 250);

  background: var(--bg);
  color: oklch(from var(--bg) 25% c h);
  border-color: oklch(from var(--bg) calc(l - 0.12) c h);
}
```

This is elegant. The border and text are clearly related to the background. But
the same absolute chroma is being carried into new lightness values, and that is
where generated UI relationships can start to drift or clip.

## OKHSL Gets Saturation Right

This is where OKHSL comes in.

OKHSL gives you a familiar shape: hue, saturation, lightness. But unlike HSL, it
is built on better OKLab/OKLCH math. Its most important feature for UI
generation is that saturation is normalized against what is possible for the
current hue and lightness.

Near white and black, where highly saturated colors cannot physically exist in
sRGB, OKHSL desaturates gracefully instead of clipping unpredictably.

**That behavior is exactly what a UI palette needs.**

```ts
// Illustrative pseudo-code, not a real library call.
// Same saturation, different lightness — OKHSL keeps it in-gamut.
okhsl({ h: 250, s: 70, l: 20 });
okhsl({ h: 250, s: 70, l: 50 });
okhsl({ h: 250, s: 70, l: 90 });
```

In OKHSL, `70%` saturation means something like:

> Use 70% of the chroma available for this hue and lightness.

That is much better for generated UI colors than:

> Use this absolute chroma value and hope it still fits.

One aside before moving on: OKHSL is limited to sRGB, and that may sound
disappointing in a world of wide-gamut displays. For core UI, I think the
limitation is often a feature. Text and background pairs are mostly about
luminance contrast, and highly saturated text, icons, and borders can feel loud
or vibrating — even tiring — when they are small and repeated across an
interface. Wide-gamut color is great for illustration, charts, and brand
moments. Core UI tokens need a conservative, predictable, in-gamut default, and
that is exactly what OKHSL gives you.

## The Bug In My Mental Model

OKHSL got me much closer to the color system I wanted.

But after using it for real UI generation, something still felt off.

At first, the problem showed up in dark mode. I could invert colors, and the
result was technically coherent, but the contrast felt weaker than expected. I
fixed that with a separate dark-mode transformation curve.

Then I noticed the same issue elsewhere: very dark colors in a light scheme also
did not move the way I expected.

That was the clue.

**The problem was not dark mode. The problem was the axis.**

**I was using lightness as if it were contrast.**

```ts
// Nice idea, weak invariant:
const dark = { ...light, l: 100 - light.l };
```

Perceptual lightness is useful. It helps colors feel evenly light or dark. But
UI contrast is about relationships between colors. Equal lightness changes do
not produce equal contrast changes across the whole range.

A simple way to see it is to compare a tiny step near black with the same tiny
step near white.

```txt
Near black: l = 0, 1, 2, 3, 4
Near white: l = 96, 97, 98, 99, 100
```

The numeric movement is the same. The contrast between adjacent steps is not.

You can also measure the failure across the whole range. Here is WCAG contrast
for a neutral ramp with equal lightness steps of `+20`:

| Lightness | vs black | vs white | +20 step × |
| --------: | -------: | -------: | ---------: |
|         0 |        1 |       21 |       1.55 |
|        20 |     1.55 |       14 |       2.09 |
|        40 |     3.23 |     6.50 |       2.05 |
|        60 |     6.64 |     3.16 |       1.86 |
|        80 |       12 |     1.70 |       1.70 |
|       100 |       21 |        1 |          — |

The numeric steps are identical, but the contrast multiplier between adjacent
steps drifts from `1.55x` up to `2.09x` and back down to `1.70x`. That drift
makes lightness a poor authoring axis for things like borders, hover states,
shadows, and dark-mode inversion.

I wanted an axis where a numeric step meant something stable for UI.

**That axis became tone.**

## OKHST

OKHST keeps hue and saturation from OKHSL, verbatim.

It replaces lightness with **tone**. And structurally, that is the whole change:
**tone is a monotonic remap of OKHSL's lightness axis.** A given tone always
maps to the same OKHSL lightness, for every hue and saturation. Every OKHST
color is a valid OKHSL color, so gamut behavior and reversibility are inherited
exactly.

What changes is what the numbers mean. Tone is a normalized logarithm of
luminance, offset by the same `0.05` used in the WCAG contrast formula:

```txt
tone(Y) = 100 * (ln(Y + 0.05) - ln(0.05)) / (ln(1.05) - ln(0.05))
```

That small connection is the whole trick. WCAG contrast is
`(Y_hi + 0.05) / (Y_lo + 0.05)`, so for neutral colors a fixed tone difference
corresponds to a fixed contrast ratio, no matter where on the scale it sits.

Here is the same measurement as before, but with equal tone steps instead of
equal lightness steps:

| Tone | vs black | vs white | +20 step × |
| ---: | -------: | -------: | ---------: |
|    0 |        1 |       21 |       1.84 |
|   20 |     1.84 |       11 |       1.84 |
|   40 |     3.38 |     6.21 |       1.84 |
|   60 |     6.21 |     3.38 |       1.84 |
|   80 |       11 |     1.84 |       1.84 |
|  100 |       21 |        1 |          — |

Each `+20` tone step multiplies contrast by exactly the same factor — `1.84x`.
That is the property plain lightness does not have.

The mental model becomes simple:

```ts
// Illustrative pseudo-code, not a real library call.
const primary = okhst({ h: 250, s: 70, t: 62 });
const hover = okhst({ h: 250, s: 70, t: 56 });
const darkPrimary = okhst({ h: 250, s: 70, t: 38 }); // 100 - 62
```

Tone movement can describe tints, shades, borders, shadows, and state changes.
Dark mode can start from one rule:

```txt
dark tone = 100 - light tone
```

That does not mean OKHST magically guarantees accessibility. _It does not._ The
uniformity is in lightness, not luminance: once saturation enters, rendered
sRGB luminance drifts, so measured contrast for chromatic colors can move away
from the neutral prediction. If you need a contrast floor, you still need to
measure WCAG or APCA against the final rendered color.

But the search space is better. The numbers are more obvious. A methodology can
say "move tone by this much" and get a predictable result, which means fewer
iterations and less guesswork for humans. It matters even more for AI-assisted
UI generation: if the axes are predictable, an AI can follow rules instead of
hallucinating a random palette and hoping it looks right.

OKHST is not a universal color science breakthrough. It is a practical
coordinate system for generating UI palettes.

The full specification is here: [github.com/tenphi/okhst](https://github.com/tenphi/okhst)

## Scheme Mapping

OKHST gives us the primitive:

```txt
hue, saturation, tone
```

But real interfaces need one more layer.

In a light scheme, I usually do not want the full `0-100` tone range. Pure black
or near-black UI colors can create unnecessary harshness. A practical light
scheme can map authored tones into something like `10-100` — asymmetric on
purpose, because pure white surfaces are exactly what people expect from a light
theme, while pure black text on them is harsher than it needs to be.

In a dark scheme, the problem flips. Very black surfaces and pure white text can
be tiring. Dark UI often feels better when it avoids both extremes, so a
practical dark scheme can invert tone and map it into something like `15-95`.

High contrast is different. There, the goal is maximum separation, so it should
usually use the full `0-100` range.

**This mapping is not part of OKHST itself. It is UI logic layered on top.**

But it is important UI logic.

It means dark mode and high contrast can become parameter changes instead of
separate hand-authored palettes. The same authored tones can be remapped for the
target scheme.

Tone flipping also needs to be local. Not every color region should invert in
dark mode. An accent area may need to keep its direction stable while the
surrounding chrome is dark-mapped.

That distinction becomes especially clear in the CSS playground I've built.

## The Pure-CSS Experiment

Once OKHST was small enough to explain, I wanted to see if it could be expressed
without a JavaScript library.

Modern Chromium-based browsers are starting to support CSS `@function`. That
means CSS can define custom functions with real calculations inside them.

So I tried the _obvious unreasonable thing_:

> Can OKHSL and OKHST conversion run directly in CSS?

It turns out, yes. The functions port OKHSL's gamut-normalized saturation and
emit a standard `oklch()` color, so nothing needs a JavaScript runtime.

I built a playground here: [tenphi.github.io/okhst](https://tenphi.github.io/okhst/)

{/* TODO: screenshot — OKHST playground at tenphi.github.io/okhst (not final) */}

The playground has two functions:

```css
/* Raw OKHST primitive: direct hue/saturation/tone conversion. */
--raw-color: --okhst(var(--seed-h), var(--seed-s), 62);

/* UI helper: maps tone into the active scheme before resolving OKHST. */
--ui-color: --ghst(var(--seed-h), var(--seed-s), 62);
```

`--okhst()` is the color-space primitive.

`--ghst()` is the scheme-aware helper. It takes the same arguments, but the
tone first passes through the current tone window and optional tone flip, and
only then reaches `--okhst()`. The name is deliberately short, because almost
every color in the playground goes through it.

The fun part is that the whole playground UI is themed this way. Changing the
hue, saturation, scheme, or tone window does not only change the swatch ramp. It
changes the interface around it. This is the slider from the first paragraph of
this post, actually working: one `hue` slider, one `saturation` slider, and the
whole interface follows.

The chrome colors are generated from the same seed:

```css
:root {
  --seed-h: 250;
  --seed-s: 70;

  --c-surface: --ghst(var(--seed-h), 4, 100);
  --c-elev: --ghst(var(--seed-h), 5, 96);
  --c-border: --ghst(var(--seed-h), 4, 86);
  --c-text: --ghst(var(--seed-h), 2, 6);
  --c-muted: --ghst(var(--seed-h), 2, 38);
  --c-accent: --ghst(var(--seed-h), var(--seed-s), 58);
}
```

One note on units: saturation and tone are both `0-100` here, matching the
percent form. Hue is degrees, and low saturations like `4` are nearly neutral.

The scheme mapping is just variables:

```css
:root {
  --tone-lo: 10;
  --tone-hi: 100;
  --tone-flip: 0;
}

html.dark {
  --tone-lo: 15;
  --tone-hi: 95;
  --tone-flip: 1;
}

html.high-contrast {
  /* --tone-flip is inherited from the active light/dark scheme. */
  --tone-lo: 0;
  --tone-hi: 100;
}

.accent-zone {
  --tone-flip: 0;
}
```

And the swatches calculate their own relationships:

```css
.swatch {
  --tone-flip: 0; /* palette itself does not invert in dark mode */
  --t: calc(var(--i) / (var(--count) - 1) * 100);
  --swatch-bg: --ghst(var(--seed-h), var(--seed-s), var(--t));

  background-color: var(--swatch-bg);
  color: --ghst(var(--seed-h), calc(var(--seed-s) * 0.4), calc(100 - var(--t)));
}
```

This is the part I like most. **The colors are not isolated values. They are
relationships:**

- the swatch tone comes from its index
- the text tone is inverted from the background tone
- the text saturation is reduced relative to the source saturation
- the palette itself opts out of dark inversion
- the surrounding UI still follows the active scheme

Rendered with those rules (precomputed, since CSS `@function` is not widely
shipped yet), the ramp looks like this:

The first version of the playground was slow, though. Dragging the hue slider
re-ran too much color math for every swatch, and an interaction could take
around a second. Hoisting the color declarations and reusing the swatch
structure helped, but not enough.

After I showed the demo to [Roman Komarov](https://kizu.dev/), he suggested a
proof-of-concept optimization for the most expensive function path,
`--okhst-smax()`, splitting its branch selection and setup work into smaller
CSS functions. That alone gave roughly a 6x speedup, and it pointed me at the
larger refactor: the expensive cusp calculation depends only on hue, so it can
be hoisted, cached as registered custom properties, and inherited by the
swatches instead of recalculated inside every color. With that, plus splitting
the OKHST and OKHSL hot paths, the same interaction went from around one second
to around 30ms.

Two design notes if you try this yourself. First, the hoisted cache depends on
hue, so a real product with several hues (primary, danger, warning, success)
needs a scope per theme where the hue-specific math is computed once and
inherited locally. Second, relationships need to be computed in place: if you
resolve a color into a global custom property too early, you lose local tone
flipping, tone windows, and scheme behavior.

In raw runtime cost, 30ms might even be acceptable for production. The real
blocker is browser support: CSS `@function` is Chromium-only today (Chrome/Edge
139+), and Firefox and Safari have not shipped it. Realistically, that means
waiting around three years, maybe more, before this can become a normal
production technique for a broad audience.

Still, that is the promise of this approach: keep the color relationships alive
in CSS instead of flattening them into static values too early. The standards
are already expressive enough to implement the model. They are just not widely
supported enough to depend on yet.

## Use Glaze Today

The pure-CSS version is the fun part because it proves the browser can express
the idea.

But if you need generated UI colors **today**, use
[Glaze](https://github.com/tenphi/glaze).

Glaze is the production-ready version of this thinking. It uses OKHST for color
generation, supports dependent colors, solves WCAG and APCA contrast, handles
light, dark, and high-contrast schemes, and exports to real design-token
formats.

That will be a separate article, because Glaze is not just a color space demo.
It is a methodology for building production UI palettes.

## Closing

The goal is not to make UIs more colorful. It is to make the boring parts of
color reliable: text that stays readable, borders that stay visible, shadows
that make sense, dark mode that does not need a second hand-authored palette,
and high contrast that does not mean starting over.

OKHST is one attempt to make those relationships easier to author. It is not
perfect or universal, but it is more predictable, which is what I wanted for UI
color generation.

And the question this post opened with — wouldn't it be great to change a whole
interface palette by moving one `hue` slider? — is no longer hypothetical. Open
the [playground](https://tenphi.github.io/okhst/), drag the slider, and watch
the boring parts follow.
