Wouldn't it be great if moving one `hue` slider could recolor an entire
interface?

Add a `saturation` slider, and you could also decide how colorful the system
should feel.

I'm not talking about a beautiful gradient or a decorative palette. I mean the
boring parts of an interface: backgrounds, text, borders, hover states, selected
states, disabled states, shadows, dark mode, and high contrast.

On paper, it sounds simple: pick a hue and saturation, then derive everything
else. The trouble starts when it has to work for every hue.

I have chased that idea through several color spaces over the years. Most of
the problems were visible and fixable. The hardest one to spot was an assumption
baked into my own mental model:

**lightness is not contrast.**

This post follows how I found that bug and built OKHST, a small coordinate
system designed around it. The CSS playground later in the article tests the
model in the browser; Glaze is the production implementation I use today.

## The Old Attempt

About seven years ago, I tried to generate UI colors at runtime. Instead of
hand-picking every token, the framework would take a small set of inputs and
derive a complete scheme. [HSLuv](https://www.hsluv.org/) made the experiment
possible.

HSLuv gave me something regular HSL did not: a perceptually useful lightness
axis. In regular HSL, `50%` lightness can mean wildly different things for
yellow, blue, red, and green. HSLuv made "make this lighter" and "make this
darker" much closer to operations a UI system could rely on.

I spent months tuning formulas around it. Eventually, tiny declarations could
color different parts of an interface while the framework kept the palette
consistent. The result was good enough for production, and it was one of the
most fun things I had built at the time.

The old demo is still online: [theme.tenphi.me](https://theme.tenphi.me)

But the system was coupled to one design system, embedded in our framework, and
tuned for one product. HSLuv was also still immature, so several inconsistencies
needed local workarounds.

The experiment worked, but it never became a reusable tool. I kept using HSLuv,
stopped shipping generated palettes, and held on to the idea that UI colors
should be computable.

## The Web Caught Up

The web has since gained OKLCH in CSS. It gives the platform a modern perceptual
color space and makes many color operations more sensible than HSL ever could.

But OKLCH is still hard to use directly as a UI palette generator.

The problem is its chroma axis. In OKLCH, `C` is absolute chroma, not the
relative "saturation" control people expect when authoring UI colors. The
maximum possible chroma changes with hue and lightness, so keeping chroma fixed
while changing lightness can push some colors outside sRGB and cause clipping.

```css
/* Same chroma, different lightness.
   Some hue/lightness combinations may clip in sRGB. */
.step-1 {
  color: oklch(35% 0.18 250);
}
.step-2 {
  color: oklch(65% 0.18 250);
}
.step-3 {
  color: oklch(85% 0.18 250);
}
```

That is manageable when defining dozens of colors manually: pick each value by
eye and adjust the outliers.

The problem appears when colors need to be related to each other, like in a
dynamic theming system. If a background, text color, border, shadow, and hover
state are all derived from one source color, clipping silently changes the
relationship you were trying to preserve. A generated ramp can look fine in the
middle and fall apart near the edges.

Modern CSS makes this very tempting:

```css
.card {
  --bg: oklch(65% 0.18 250);

  background: var(--bg);
  color: oklch(from var(--bg) 25% c h);
  border-color: oklch(from var(--bg) calc(l - 0.12) c h);
}
```

The relationship is elegant, but the same absolute chroma follows the
background into new lightness values. That is where a generated palette can
start to drift or clip. What I needed instead was a saturation axis relative to
the maximum chroma available at each hue and lightness.

## OKHSL Gets Saturation Right

OKHSL provides exactly that axis. It has the familiar shape of hue, saturation,
and lightness, but builds it on OKLab/OKLCH math. Most importantly for UI
generation, its saturation is normalized against what is possible for the
current hue and lightness.

Near white and black, where highly saturated colors cannot physically exist in
sRGB, OKHSL desaturates gracefully instead of clipping unpredictably.

That graceful behavior is what a generated UI palette needs.

```ts
// Illustrative pseudo-code, not a real library call.
// Same saturation, different lightness — OKHSL keeps it in-gamut.
okhsl({ h: 250, s: 70, l: 20 });
okhsl({ h: 250, s: 70, l: 50 });
okhsl({ h: 250, s: 70, l: 90 });
```

In OKHSL, `70%` saturation means something like:

> Use 70% of the chroma available for this hue and lightness.

That is much better for generated UI colors than:

> Use this absolute chroma value and hope it still fits.

OKHSL is limited to sRGB, which may sound disappointing in a world of wide-gamut
displays. For core UI, I often find that constraint useful: repeated text,
icons, and borders benefit from a conservative, in-gamut default. Wide-gamut
color still has a natural place in illustrations, decorations, and brand moments.

## The Bug In My Mental Model

With hue and saturation behaving well, OKHSL got me much closer to the system I
wanted. But the palettes still felt wrong in places.

Dark mode exposed the problem first. Inverting lightness produced a coherent
palette, but the contrast felt weaker than expected, so I added a separate
transformation curve. Then I noticed that very dark colors in a light scheme
also moved differently from what I expected. The workaround had treated the
symptom: I was using lightness as if it were contrast.

```ts
// Nice idea, weak invariant:
const dark = { ...light, l: 100 - light.l };
```

Perceptual lightness is useful because it helps colors feel evenly light or
dark. UI contrast, however, describes the relationship between two colors.
Equal lightness changes do not produce equal contrast changes across the whole
range.

A simple way to see it is to compare a tiny step near black with the same tiny
step near white.

```txt
Near black: l = 0, 1, 2, 3, 4
Near white: l = 96, 97, 98, 99, 100
```

The numeric movement is the same. The contrast between adjacent steps is not.

You can also measure the failure across the whole range. Here is WCAG contrast
for a neutral ramp with equal lightness steps of `+20`:

| Lightness | vs black | vs white | +20 step × |
| --------: | -------: | -------: | ---------: |
|         0 |        1 |       21 |       1.55 |
|        20 |     1.55 |       14 |       2.09 |
|        40 |     3.23 |     6.50 |       2.05 |
|        60 |     6.64 |     3.16 |       1.86 |
|        80 |       12 |     1.70 |       1.70 |
|       100 |       21 |        1 |          — |

The numeric steps are identical, but the contrast multiplier between adjacent
steps drifts from 1.55× up to 2.09× and back down to 1.70×. That drift
makes lightness a poor authoring axis for things like borders, hover states,
shadows, and dark-mode inversion.

I needed an authoring axis where a numeric step described a stable change in UI
contrast. That axis became **tone**.

## OKHST

OKHST keeps OKHSL's hue and saturation and replaces lightness with tone.
Structurally, tone is a monotonic remap of the OKHSL lightness axis: a given
tone always maps to the same OKHSL lightness at every hue and saturation. Every
OKHST color is therefore a valid OKHSL color, inheriting its gamut behavior and
reversibility.

What changes is what the numbers mean. Tone is a normalized logarithm of
luminance, offset by the same `0.05` used in the WCAG contrast formula:

```txt
tone(Y) = 100 * (ln(Y + 0.05) - ln(0.05)) / (ln(1.05) - ln(0.05))
```

That connection gives tone its useful behavior. WCAG contrast is
`(Y_hi + 0.05) / (Y_lo + 0.05)`, so for neutral colors a fixed tone difference
corresponds to a fixed contrast ratio, no matter where on the scale it sits.

Here is the same measurement as before, but with equal tone steps instead of
equal lightness steps:

| Tone | vs black | vs white | +20 step × |
| ---: | -------: | -------: | ---------: |
|    0 |        1 |       21 |       1.84 |
|   20 |     1.84 |       11 |       1.84 |
|   40 |     3.38 |     6.21 |       1.84 |
|   60 |     6.21 |     3.38 |       1.84 |
|   80 |       11 |     1.84 |       1.84 |
|  100 |       21 |        1 |          — |

Each `+20` tone step multiplies contrast by exactly the same factor — 1.84×.
That is the property plain lightness does not have.

Tone lets states be authored as deltas instead of independent colors:

```ts
// Illustrative pseudo-code, not a real library call.
const primaryTone = 62;
const hoverDelta = 6;

const primary = okhst({ h: 250, s: 70, t: primaryTone });
const hover = okhst({ h: 250, s: 70, t: primaryTone - hoverDelta });
```

The same pattern can describe tints, shades, borders, and shadows. Scheme
mapping belongs to the layer above the color primitive; dark mode can begin
with one transformation:

```ts
const darkTone = (tone: number) => 100 - tone;

const darkPrimary = okhst({
  h: 250,
  s: 70,
  t: darkTone(primaryTone),
});
```

OKHST does not guarantee accessibility. The fixed relationship is exact for
neutral colors; once saturation enters, rendered sRGB luminance can drift from
that neutral prediction. A required contrast floor still has to be checked with
WCAG or APCA against the final rendered color pair.

The search space is still much better. A methodology can say "move tone by this
much" and produce a consistent starting point, reducing iteration and guesswork.
That matters even more for AI-assisted UI generation: well-behaved axes let an
AI follow explicit rules instead of inventing a palette and hoping it works.

OKHST is not a universal color science breakthrough. It is a practical
coordinate system for generating UI palettes.

The full specification is here: [github.com/tenphi/okhst](https://github.com/tenphi/okhst)

## Scheme Mapping

OKHST gives us the primitive:

```txt
hue, saturation, tone
```

Interfaces add scheme logic on top of that primitive: tone windows, tone
deltas, and local tone flips.

A **tone window** maps the authored `0-100` range into the endpoints a scheme
should actually use. For a light scheme, I might use `10-100`: pure white
surfaces remain available, while text avoids unnecessarily harsh black. A dark
scheme can invert tone and map it into `15-95`, avoiding both black surfaces and
pure white text. High contrast usually opens the window to the full `0-100`
range.

Endpoints alone are not enough. A **tone delta** defines how far one color sits
from its parent: a border might be `14` tones away from its surface, for
example. Opening the window preserves that distance, so a subtle border remains
subtle. High contrast also needs larger deltas, pulling borders, muted text, and
accents further from the colors beneath them.

Because tone deltas have stable contrast behavior, scheme behavior reduces to
parameters: a tone window plus a set of deltas. The same authored palette can
then serve light, dark, and both high-contrast variants instead of becoming four
separately maintained palettes.

Tone inversion is local as well. The surrounding chrome may follow the dark
scheme while an accent region keeps its original direction. This mapping is UI
logic layered on top of OKHST, not part of the color space itself.

The CSS playground makes those separate layers concrete.

## The Pure-CSS Experiment

The playground tests whether the OKHST primitive and its scheme logic can stay
live in CSS instead of being flattened into static values. Modern
Chromium-based browsers are starting to support CSS `@function`, which allows
custom functions with real calculations. That made an unreasonable experiment
possible:

> Can OKHSL and OKHST conversion run directly in CSS?

The answer is yes. The functions port OKHSL's gamut-normalized saturation and
emit standard `oklch()` colors without a JavaScript runtime.

I built a playground here: [tenphi.github.io/okhst](https://tenphi.github.io/okhst/)

{/* TODO: screenshot — OKHST playground at tenphi.github.io/okhst (not final) */}

The implementation exposes two functions:

```css
/* Raw OKHST primitive: direct hue/saturation/tone conversion. */
--raw-color: --okhst(var(--seed-h), var(--seed-s), 62);

/* UI helper: maps tone into the active scheme before resolving OKHST. */
--ui-color: --ghst(var(--seed-h), var(--seed-s), 62);
```

`--okhst()` is the color-space primitive. `--ghst()` is the scheme-aware helper:
it passes tone through the current window and optional flip before calling
`--okhst()`. Its short name is useful because nearly every playground color
goes through it.

The playground UI uses the same system as its swatches. Changing hue,
saturation, scheme, or tone window also changes the interface around the ramp.
The two sliders from the opening are now working: hue chooses the color family,
saturation controls its intensity, and the UI follows.

The chrome colors are generated from the same seed. Each one is authored as a
tone delta from the surface — the same parent-relative steps from the previous
section:

```css
:root {
  --seed-h: 250;
  --seed-s: 70;

  /* tone deltas: how far each color steps away from the surface */
  --d-elev: 4;
  --d-border: 14;
  --d-text: 94;
  --d-muted: 62;
  --d-accent: 42;

  --c-surface: --ghst(var(--seed-h), 4, 100);
  --c-elev: --ghst(var(--seed-h), 5, calc(100 - var(--d-elev)));
  --c-border: --ghst(var(--seed-h), 4, calc(100 - var(--d-border)));
  --c-text: --ghst(var(--seed-h), 2, calc(100 - var(--d-text)));
  --c-muted: --ghst(var(--seed-h), 2, calc(100 - var(--d-muted)));
  --c-accent: --ghst(var(--seed-h), var(--seed-s), calc(100 - var(--d-accent)));
}
```

Saturation and tone both use the `0-100` percent form here. Hue uses degrees,
and low saturations such as `4` are nearly neutral.

Scheme behavior reduces to variables:

```css
:root {
  --tone-lo: 10;
  --tone-hi: 100;
  --tone-flip: 0;
}

html.dark {
  --tone-lo: 15;
  --tone-hi: 95;
  --tone-flip: 1;
}

html.high-contrast {
  /* --tone-flip is inherited from the active light/dark scheme. */
  --tone-lo: 0;
  --tone-hi: 100;

  /* wider steps: relationships pull apart, not just the extremes */
  --d-elev: 6;
  --d-border: 22;
  --d-text: 100;
  --d-muted: 72;
  --d-accent: 50;
}

.accent-zone {
  --tone-flip: 0;
}
```

Because the chrome colors derive from deltas, overriding these variables
implements high contrast for light and dark at once; tone inversion remains a
separate layer.

The swatches define their own local rules:

```css
.swatch {
  --tone-flip: 0; /* palette itself does not invert in dark mode */
  --t: calc(var(--i) / (var(--count) - 1) * 100);
  --swatch-bg: --ghst(var(--seed-h), var(--seed-s), var(--t));

  background-color: var(--swatch-bg);
  color: --ghst(var(--seed-h), calc(var(--seed-s) * 0.4), calc(100 - var(--t)));
}
```

These colors are not isolated values; the rules keep them connected:

- the swatch tone comes from its index
- the text tone is inverted from the background tone
- the text saturation is reduced relative to the source saturation
- the palette itself opts out of dark inversion
- the surrounding UI still follows the active scheme

The playground reads final colors with `getComputedStyle`, but CSS performs

The first version was slow because every swatch repeated too much color math.
After I showed it to [Roman Komarov](https://kizu.dev/), his proof-of-concept
split setup work from the most expensive function path and produced roughly a
6x speedup. That led to a broader refactor: cache the hue-dependent cusp
calculation, inherit it through the swatches, and separate the OKHST and OKHSL
hot paths. Slider interactions fell from around one second to around 30ms. In a
product with several semantic hues, each hue needs its own cache scope; the
final color rules must also resolve locally to preserve tone windows and flips.

In raw runtime cost, 30ms might even be acceptable for production. The real
blocker is browser support: CSS `@function` is Chromium-only today (Chrome/Edge
139+), and Firefox and Safari have not shipped it. Realistically, that means
waiting around three years, maybe more, before this can become a normal
production technique for a broad audience.

The experiment still proves that CSS can preserve palette rules instead of
flattening them into static values. The platform is expressive enough for the
model, just not widely supported enough to depend on yet.

## The Hue Slider Loophole

This blog already ships one narrow slice of the idea in plain CSS across modern
browsers.

The palette icon in the top bar opens a hue slider. Dragging it rotates the
site's text, surfaces, borders, and accents without `@function` or JavaScript
color math. Underneath is the same pattern I called fragile earlier:

```css
--text-color: oklch(0.201 0.0014 var(--primary-hue));
```

The apparent contradiction disappears because every rotating color obeys one
hard restriction: it lives in a **pastel space**. Chroma stays below the level
that fits inside sRGB for every hue at that lightness. Within that boundary,
hue can rotate without clipping or overshooting.

The shortcut has three costs:

- **Saturation is capped.** Pastel space excludes vivid colors.
- **Only hue remains dynamic.** Tone windows, state changes, and deltas must be
  baked into static values.
- **Each pastel limit must be precomputed.** Otherwise, every color needs a CSS
  function or an impractically large inline calculation.

This blog performs that work at build time, leaving only hue as a custom
property. The loophole is useful but narrow: one dynamic axis, muted colors,
and precomputation behind the scenes. A production system has to provide that
precomputation.

## Use Glaze Today

That production system is [Glaze](https://github.com/tenphi/glaze). It applies
OKHST and precomputes the palette rules that browsers cannot yet evaluate
broadly at runtime. It supports dependent colors, solves WCAG and APCA
contrast, handles light, dark, and high-contrast schemes, and exports standard
design-token formats.

This blog is a live demo. Glaze generates both the hue-rotating UI palette and
the syntax-highlighting palette, which can skip the pastel restriction because
it never rotates. The dark and high-contrast modes behind the same palette icon
are scheme remaps precomputed at build time: parameter changes rather than
hand-authored alternate palettes. [tasty.style](https://tasty.style) also uses
Glaze to generate its colors and provide dark and high-contrast theme variants.

The approach also scales beyond a single site. [Cube UI
Kit](https://github.com/cube-js/cube-ui-kit), an open-source React library with
more than 100 components that powers [Cube](https://cube.dev/) products, uses
Glaze to generate its dark scheme, with high contrast on the way. One set of
palette rules drives the entire component library.

Glaze deserves its own article: beyond implementing a color space, it provides
a methodology for building production UI palettes.

## Closing

The goal is not to make UIs more colorful. It is to make the boring parts of
color reliable.

Normalized saturation keeps hue and lightness changes inside a usable gamut.
Tone turns movement along the lightness axis into regular contrast steps.
Scheme mapping builds those properties into reusable rules for states, dark
mode, and high contrast.

OKHST is not perfect or universal, but it gives UI color generation a more
useful set of coordinates. The question from the opening is no longer
hypothetical: open the palette icon in the top bar, drag the `hue` slider, and
watch the boring parts follow.
